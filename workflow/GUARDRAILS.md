# 边界守卫（GUARDRAILS）

## 机制说明

这是每个 Skill 执行时自动加载的核心机制。边界守卫确保 AI 在正确的阶段做正确的事，防止越界操作导致混乱。

## 核心规则

**每个 Skill 只做自己阶段的事。** 需求阶段只聊需求，设计阶段只出方案，编码阶段才写代码。

当 AI 想要越界执行不属于当前阶段的操作时，必须直接拦住，明确告知用户：

> **现在不是做这个的时候。**

## 各阶段硬边界

### 项目级阶段（第一层）

| 阶段 | 能做的事 | 绝对禁止 |
|------|---------|---------|
| **需求澄清阶段** | 提问、倾听、梳理需求 | 禁止写代码、禁止讨论技术实现细节 |
| **产品概述阶段** | 定义产品定位、功能范围、用户场景 | 禁止写代码、只做产品层规划 |
| **技术栈选型阶段** | 分析对比、给出选型建议和理由 | 只做选型决策，不写任何代码 |
| **目录结构阶段** | 设计项目目录组织方式 | 只设计结构，不创建文件或写代码 |
| **开发规范阶段** | 制定编码风格、命名规范、Git规范等 | 只定规范，不写代码 |
| **路线图规划阶段** | 排定开发优先级和时间规划 | 只做规划，不写代码 |
| **项目初始化阶段** | 按文档执行初始化命令 | 严格按文档执行，不添加文档外的任何内容 |

### 功能级阶段（第二层）

| 阶段 | 能做的事 | 绝对禁止 |
|------|---------|---------|
| **功能需求澄清** | 深入理解功能细节、确认验收标准 | 禁止写代码、禁止讨论技术实现方案 |
| **技术方案设计** | 出架构图、接口设计、数据模型设计 | 只出方案文档，不写一行代码 |
| **任务规划** | 将方案拆分为可执行的编码任务 | 只拆任务列表，不写代码 |
| **编码实现** | 按任务规划逐个完成编码任务 | 不修改已确认的需求，不擅自加功能 |

### 迭代与修复阶段

| 阶段 | 能做的事 | 绝对禁止 |
|------|---------|---------|
| **功能迭代变更** | 分析变更影响、评估工作量、更新规划 | 只做分析和规划，**绝对不写代码** |
| **BUG修复** | 复现问题、定位根因、编写修复代码 | **没复现不准改代码**，禁止猜测性修改 |

## 越界拦截话术

当检测到越界行为时，使用以下格式回应：

```
⛔ 边界守卫拦截

当前阶段：[阶段名称]
你试图做的：[越界操作]
原因：[该操作属于哪个阶段]

建议：先完成当前阶段，再进入 [正确阶段]。
使用命令：/[正确阶段对应的skill名称]
```

## 核心原则

**在错误阶段做错误的事，不是帮忙，是帮倒忙。**

- 需求没清楚就写代码 → 写出来的东西没人要
- 方案没设计好就动手 → 后面全是返工
- 任务没拆好就开干 → 做到一半发现方向错了
- BUG没复现就改修 → 可能引入新问题
- 变更没分析就写代码 → 牵一发而动全身

边界守卫的存在，是为了让每一步都走得踏实，让最终交付的质量有保障。
