# AI 驱动开发工作流

## 工作流全景图

```
┌─────────────────────────────────────────────────────────────┐
│                    第一层：项目级（启动时用一次）                  │
│                                                             │
│  ①需求澄清 → ②产品概述 → ③技术栈选型 → ④目录结构              │
│       → ⑤开发规范 → ⑥路线图规划 → ⑦项目初始化                 │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                    第二层：功能级（反复使用）                     │
│                                                             │
│  ⑧功能需求澄清 → ⑨技术方案设计 → ⑩任务规划 → ⑪编码实现        │
│                                                             │
│  迭代路径：⑫功能迭代变更 → ⑪编码实现                          │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                    第三层：通用（随时使用）                      │
│                                                             │
│  ⑬BUG修复                                                   │
└─────────────────────────────────────────────────────────────┘
```

### 第一层：项目级

项目从零启动时走一遍，产出项目基础设施文档，后续不再重复。

| 步骤 | 说明 | 产出 |
|------|------|------|
| ① 需求澄清 | 深入理解用户要做什么、为什么做、给谁做 | 需求理解文档 |
| ② 产品概述 | 定义产品定位、功能范围、用户画像 | `specs/产品概述.md` |
| ③ 技术栈选型 | 根据需求选择最合适的技术方案 | `specs/技术栈.md` |
| ④ 目录结构 | 设计清晰的项目目录组织 | `specs/项目结构.md` |
| ⑤ 开发规范 | 统一编码风格和协作规范 | `specs/开发规范.md` |
| ⑥ 路线图规划 | 排定开发阶段和优先级 | `specs/开发路线图.md` |
| ⑦ 项目初始化 | 按文档执行项目搭建 | 可运行的项目骨架 |

### 第二层：功能级

每个功能开发都走这个循环，迭代时从功能迭代变更重新进入。

| 步骤 | 说明 | 产出 |
|------|------|------|
| ⑧ 功能需求澄清 | 深入理解单个功能的细节和验收标准 | `specs/features/[功能名]/需求.md` |
| ⑨ 技术方案设计 | 出架构图、接口设计、数据模型 | `specs/features/[功能名]/方案.md` |
| ⑩ 任务规划 | 将方案拆分为可执行的编码任务 | `specs/features/[功能名]/任务.md` |
| ⑪ 编码实现 | 按任务列表逐个完成编码 | 功能代码 + 测试 |
| ⑫ 功能迭代变更 | 分析变更影响、评估工作量、更新规划 | 更新后的需求/方案/任务文档 |

### 第三层：通用

随时可调用，不受项目级或功能级流程约束。

| 步骤 | 说明 | 产出 |
|------|------|------|
| ⑬ BUG修复 | 复现 → 定位 → 修复 → 验证 | 修复代码 + 复现记录 |

## 核心机制

本工作流依赖两个核心机制，每个 Skill 执行时自动加载：

### 边界守卫（GUARDRAILS）

详见 [GUARDRAILS.md](./GUARDRAILS.md)

确保 AI 在正确的阶段做正确的事。每个 Skill 只做自己阶段的事，越界行为会被直接拦截。需求阶段只聊需求，设计阶段只出方案，编码阶段才写代码。

### 项目上下文协议（PROJECT-CONTEXT）

详见 [PROJECT-CONTEXT.md](./PROJECT-CONTEXT.md)

确保 AI 在执行任何任务前，已读取 `specs/` 目录下的所有文档，建立完整的项目认知。文档在，记忆就在，开新窗口不怕丢上下文。

## 调用方式

```
/skill-name 参数
```

示例：

```
/req-clarify 我想做一个在线教育平台
/tech-design 用户认证模块
/code-impl 任务3：实现登录接口
/bug-fix 登录后偶尔跳转到404页面
```
