# 项目上下文协议（PROJECT-CONTEXT）

## 机制说明

这是每个 Skill 执行时自动加载的核心机制。项目上下文协议确保 AI 在执行任何任务前，已经建立对项目的完整认知，避免脱离项目实际情况的盲目操作。

## 核心规则

**每个 Skill 执行前，AI 必须先读取 `specs/` 目录下的所有文档，建立对项目的完整认知。不读完不许开始干活。**

## 必读文档清单

### 项目级文档（项目启动后生成）

| 文档路径 | 内容说明 | 生成阶段 |
|---------|---------|---------|
| `specs/产品概述.md` | 产品定位、目标用户、核心功能、业务边界 | 产品概述阶段 |
| `specs/技术栈.md` | 技术选型决策、框架版本、工具链 | 技术栈选型阶段 |
| `specs/项目结构.md` | 目录组织、模块划分、文件命名规范 | 目录结构阶段 |
| `specs/开发规范.md` | 编码风格、命名规范、Git规范、提交规范 | 开发规范阶段 |
| `specs/开发路线图.md` | 开发阶段划分、优先级排定、里程碑 | 路线图规划阶段 |

### 功能级文档（功能开发时生成）

| 文档路径 | 内容说明 | 生成阶段 |
|---------|---------|---------|
| `specs/features/` 目录下所有功能文档 | 每个功能的详细需求、技术方案、任务拆分 | 功能开发各阶段 |

## 读取流程

```
Skill 被调用
    ↓
检查 specs/ 目录是否存在
    ↓
存在 → 逐个读取所有文档 → 建立项目认知 → 开始执行 Skill 任务
不存在 → 提示用户先完成前置步骤
```

## 文档缺失处理

当必读文档不存在时，AI 必须停止执行当前 Skill，并向用户明确提示：

```
⚠️ 项目上下文缺失

当前缺少以下文档：
- [列出缺失的文档]

这些文档是执行 [当前Skill名称] 的前置条件。

请先完成以下步骤：
1. [提示用户需要先执行的Skill或操作]
2. [提示用户需要先执行的Skill或操作]

完成后再回来继续。
```

## 文档优先级

当多个文档存在信息冲突时，按以下优先级处理：

1. `specs/产品概述.md` — 产品定义是最高优先级
2. `specs/开发规范.md` — 规范约束次之
3. `specs/技术栈.md` — 技术选型再次之
4. `specs/项目结构.md` — 结构约定
5. `specs/开发路线图.md` — 规划安排
6. `specs/features/` — 具体功能文档

## 核心价值

**文档在，记忆就在——开新窗口不怕丢上下文。**

所有项目关键信息都以文档形式持久化在 `specs/` 目录中：

- 换一个对话窗口，AI 重新读取文档即可恢复完整认知
- 换一个开发者，看文档就能了解项目全貌
- 项目中断后重启，从文档接续而非从零开始
- 任何决策都有据可查，不会凭空遗忘

这就是为什么每个 Skill 执行前必须先读文档——文档是项目记忆的唯一可靠来源。
