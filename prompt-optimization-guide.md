# 提示词优化指南 - API 缓存命中率 + Token 消耗最低

> **适用场景**：优化 AI 编程工具的 API 调用效率，降低成本，提升响应速度
> **兼容工具**：Cursor / Windsurf / Trae / GitHub Copilot / Claude Code / 任何基于 LLM API 的编程工具
> **版本**：v1.0 | **更新日期**：2026-06-16

## 一、API 缓存命中率优化

### 1.1 缓存原理

主流 AI API（Claude/Anthropic、OpenAI、Google Gemini）都支持**前缀缓存（Prefix Caching）**：

- API 会对请求的**前缀部分**进行缓存
- 当后续请求的前缀与缓存匹配时，直接命中缓存，跳过重复计算
- 缓存命中时：响应更快、成本更低（通常节省 50%-90% 的输入 Token 费用）

### 1.2 核心优化策略

#### 策略1：静态内容前置

```
❌ 错误示范（动态内容在前）：
[用户当前问题] + [系统规则] + [项目上下文]

✅ 正确示范（静态内容在前）：
[系统规则] + [项目上下文] + [用户当前问题]
```

**原理**：系统规则和项目上下文在多轮对话中基本不变，放在前面可以最大化缓存命中。

#### 策略2：System Prompt 结构化分层

将 System Prompt 按稳定性从高到低排列：

```
第一层：核心身份（几乎不变）
  → "你是一名全栈开发专家..."

第二层：开发规范（偶尔变更）
  → 编码风格、命名规范、技术栈

第三层：项目上下文（按阶段更新）
  → specs/ 目录下的文档摘要

第四层：当前任务（每次不同）
  → 具体要做什么
```

#### 策略3：文档引用代替文档复制

```
❌ 错误示范（复制全文）：
请参考以下需求文档：
[粘贴500行需求文档全文]

✅ 正确示范（引用+摘要）：
请参考 specs/需求文档.md（已读取）。
本次任务涉及的功能：F001用户登录、F002注册。
关键约束：必须支持微信登录。
```

#### 策略4：统一 System Prompt 模板

为项目创建固定的 System Prompt 模板，所有对话共用：

```
# 项目：{项目名}
# 技术栈：{技术栈}
# 规范：{编码规范摘要}
# 上下文：{项目文档路径列表}
# 当前阶段：{阶段名}
```

模板不变部分（项目名、技术栈、规范）占 80%+，只有当前阶段和任务变化，缓存命中率极高。

### 1.3 各平台缓存机制对比

| 平台 | 缓存类型 | 最小前缀长度 | 缓存时长 | 节省比例 |
|------|---------|-------------|---------|---------|
| Anthropic(Claude) | 自动前缀缓存 | 1024 tokens | 5分钟 | 输入Token费用-90% |
| OpenAI(GPT) | 自动前缀缓存 | 1024 tokens | 5-10分钟 | 输入Token费用-50% |
| Google(Gemini) | 上下文缓存 | 自定义 | 自定义(最长1小时) | 输入Token费用-75% |

### 1.4 实操配置

#### Cursor 配置

```

## 二、Token 消耗最小化

### 2.1 核心原则

| 原则 | 说明 | 节省效果 |
|------|------|---------|
| 精准指令 | 说清楚要什么，不说废话 | 30-50% |
| 结构化输出 | 用表格/列表代替段落 | 20-40% |
| 增量变更 | 只输出变更部分 | 50-80% |
| 上下文管理 | 只传必要上下文 | 30-60% |
| 分阶段执行 | 一次只做一个任务 | 20-40% |

### 2.2 精准指令技巧

```
❌ 冗长指令（~200 tokens）：
你好，我想请你帮我做一个用户登录的功能。这个功能需要支持邮箱登录和微信登录，
邮箱登录需要验证码，微信登录需要扫码。登录成功后要跳转到首页，如果失败要显示
错误提示。另外还需要记住登录状态，下次打开不用重新登录。请帮我实现这个功能。

✅ 精准指令（~60 tokens）：
实现用户登录功能：
- 邮箱登录（验证码）+ 微信扫码登录
- 成功→首页，失败→错误提示
- 记住登录状态（7天）
参考：specs/features/登录/方案.md
```

### 2.3 结构化输出模板

要求 AI 使用固定格式输出，减少冗余描述：

```
输出要求：
- 代码：只输出变更的函数/组件，标注文件路径
- 说明：用表格，不用段落
- 问题：用列表，标注严重程度[P0/P1/P2]
- 不输出：未修改的代码、过渡性语言、重复信息
```

### 2.4 增量变更策略

```
❌ 全量输出：
[输出整个500行文件]

✅ 增量输出：
文件：src/components/LoginForm.tsx
变更：handleLogin 函数
- 添加微信登录分支
- 修改错误处理逻辑

```typescript
// 只输出变更的函数
const handleLogin = async (method: 'email' | 'wechat') => {
  if (method === 'wechat') {
    // 新增：微信扫码登录逻辑
    const qrCode = await getWechatQR();
    setQrData(qrCode);
  } else {
    // 原有邮箱登录逻辑（未修改，不输出）
  }
};
```
```

### 2.5 上下文管理策略

#### 规则1：按需加载，不全量读取

```
❌ 每次都读取所有文件：
请读取项目中所有文件，然后帮我修改登录功能。

✅ 只读取必要文件：
请读取以下文件后帮我修改登录功能：
1. specs/features/登录/方案.md（技术方案）
2. src/components/LoginForm.tsx（要修改的文件）
3. src/api/auth.ts（相关API）
```

#### 规则2：文档摘要代替全文

```
❌ 传递全文（5000 tokens）：
[粘贴完整的需求文档]

✅ 传递摘要（200 tokens）：
需求摘要（详见 specs/需求文档.md）：
- P0功能：登录、注册、首页
- 技术约束：必须支持微信登录
- 当前任务：实现登录功能
```

#### 规则3：新窗口清理上下文

```
当上下文累积超过以下阈值时，开新窗口：
- 简单任务：对话超过10轮
- 中等任务：对话超过6轮
- 复杂任务：每个任务开新窗口

新窗口只需：
1. 读取 specs/ 下的文档（自动恢复项目认知）
2. 说明当前任务
3. 开始工作
```

### 2.6 对话压缩技巧

在长对话中，定期让 AI 压缩上下文：

```
请用以下格式总结我们目前的进展：
- 已完成：[列表]
- 当前任务：[描述]
- 待解决问题：[列表]
- 下一步：[描述]

然后将总结保存到 specs/进度.md，我们开新窗口继续。
```

## 三、跨平台兼容性

### 3.1 通用提示词结构

以下结构在所有主流 AI 编程工具中都能正常工作：

```
# 角色定义
你是一名[角色]，负责[职责]。

# 硬规则
1. [规则1]
2. [规则2]

# 执行步骤
1. [步骤1]
2. [步骤2]

# 输出格式
[格式模板]

# 约束
- [约束1]
- [约束2]
```

### 3.2 各平台配置文件对照

| 平台 | 配置文件 | 放置位置 | 格式 |
|------|---------|---------|------|
| Cursor | `.cursorrules` | 项目根目录 | 纯文本 |
| Windsurf | `.windsurfrules` | 项目根目录 | 纯文本 |
| Trae | `.trae/rules/project_rules.md` | 项目目录 | Markdown |
| Claude Code | `CLAUDE.md` | 项目根目录 | Markdown |
| GitHub Copilot | `.github/copilot-instructions.md` | 项目根目录 | Markdown |
| Cline | `.clinerules` | 项目根目录 | 纯文本 |
| Aider | `.aider.conf.yml` | 项目根目录 | YAML |

### 3.3 统一配置文件生成

创建一个通用配置，然后为每个平台生成对应格式：

```bash
# 通用规则内容（保存为 rules/common.md）
# 然后用脚本或手动复制到各平台配置文件
```

通用规则内容模板：

```
# 项目规则

## 技术栈
- 前端：[框架] + [语言] + [样式方案]
- 后端：[框架] + [语言]
- 数据库：[类型]
- 部署：[平台]

## 编码规范
- 命名：camelCase变量 / PascalCase组件 / kebab-case文件
- 样式：[方案]优先
- 状态管理：[方案]
- API调用：统一使用封装函数
- 错误处理：统一错误处理模式

## 开发流程
- 先读 specs/ 文档再动手
- 一次只做一个任务
- 每个任务完成写测试
- 不自由发挥，按文档执行

## 输出要求
- 代码只输出变更部分
- 说明用表格，不用段落
- 不输出未修改的代码
- 不使用过渡性语言
```

### 3.4 平台特定优化

#### Cursor 优化
- 使用 `@file` 引用文件而非粘贴内容
- 使用 `@folder` 引用目录
- 在 `.cursorrules` 中放置项目级规则
- 使用 Composer 模式处理多文件变更

#### Windsurf 优化
- 使用 Cascade 模式进行多步骤任务
- 在 `.windsurfrules` 中定义项目规则
- 利用 Memories 功能保存常用上下文

#### Trae 优化
- 使用 Skill 系统触发预定义工作流
- 在 `.trae/rules/` 中定义项目规则
- 利用多窗口管理不同阶段的上下文

#### Claude Code 优化
- 使用 `CLAUDE.md` 定义项目规则
- 利用 `@file` 引用文件
- 使用 `/init` 命令初始化项目规则
